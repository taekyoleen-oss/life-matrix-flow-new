# Editor UX Review — 편집기 사용성 검토 방법론

## 여정 기반 검토

기능 목록이 아니라 사용자가 실제 거치는 순서로 마찰을 찾는다. 각 단계에서 "다음에 뭘 해야 할지 명확한가?"를 묻는다.

### 여정 단계와 점검 포인트

1. **모듈 추가** (Toolbox→Canvas): 모듈이 뭘 하는지 추가 전에 아는가(설명/툴팁)? 한/영 용어 혼란은?
2. **포트 연결:** 어떤 포트가 어디에 연결되는지 시각적으로 명확한가? 비호환 연결을 막거나 경고하는가? 연결 방향이 직관적인가?
3. **파라미터 입력** (모달들): 무엇을 입력해야 하는지 명확한가? 컬럼명·수식을 **수동 타이핑**하게 해서 오타를 유발하는가(→드롭다운 권장)? 필수/선택 구분, 검증, 예시·기본값이 있는가? 저장/취소 동작이 일관적인가?
4. **실행** (개별/전체): 지금 무엇이 실행 중/대기/완료/실패인지 보이는가(`ModuleStatus`)? 실패 시 어느 모듈이 왜 실패했는지 즉시 아는가?
5. **결과 확인** (Code/Terminal/Preview/Report): 결과를 이해할 수 있는가? 코드·터미널·미리보기 간 전환이 자연스러운가?

## 핵심 점검 축

- **단계별 진행 피드백:** 다단계 산출에서 현재 위치·완료도·다음 필요사항이 보이는가. 미연결/미완료 모듈이 "왜 안 되는지" 알려주는가.
- **입력오류 예방 > 사후 에러:** 잘못 입력 가능한 지점을 찾아 예방책(드롭다운/검증/자동완성)을 제안. 에러 메시지가 사용자 언어로, 고칠 위치를 알려주는가.
- **일관성:** 모달 간 버튼 배치, 용어, 저장/취소, 다크모드.
- **초심자:** 보험계리 비전문가도 이해 가능한가. 용어 설명 없이 쓰지 않기.

## 개선안 작성 기준

"개선 필요"가 아니라 구체적으로: 대상 컴포넌트(`file:line`) · 현재 동작 · 제안 동작 · 사용자에게 주는 이득. 우선순위 P0(깨짐/막힘)/P1(큰 마찰)/P2(개선) 부여. 가능하면 Playwright로 실제 화면 동작 확인 후 작성.

## 출력

`_workspace/02_ux_review.md`: `## 여정별 마찰 목록`(단계·[P0/P1/P2]·현상·사용자영향·개선안·`file:line`), `## Quick Wins`(빠른 개선), `## 구조적 개선`.

## 원칙

주관(미관)과 객관(깨진 동작) 구분. 실측 못 하면 "실측 미확인" 명시. sync-architect의 새 편집 UI는 이 기준으로 별도 검토.
